# Решения по Yakov Viewer

Этот файл фиксирует текущие решения из обсуждения, простым языком.

## Как работаем

Пользователь часто надиктовывает сообщения голосом в шумной обстановке. Распознавание может путать слова, названия доменов, файлы и технические термины.

Практическое правило: перед опасными или внешне видимыми действиями нужен короткий бриф на 2-4 строки. Если фраза выглядит странно или противоречит уже принятому направлению, нужно уточнить, а не исполнять дословно.

## Главная идея

Yakov Viewer - это публичный сайт-портфолио фотографа Якова Шмоля и одновременно основа для большой общей фотоархивной системы.

Админка - это важная самостоятельная часть проекта. Она нужна не только для управления публичным сайтом, но и как private-хранилище: там могут лежать сеты, альбомы и фотографии, которые никогда не будут опубликованы.

Эта же база и это же хранилище должны в будущем использоваться и для Logjamming, чтобы не загружать один и тот же архив дважды.

## Структура архива

Базовая структура:

```txt
Сет
  Альбом
    Фотография
      Версии файла

Коллекция
  Выбранные фотографии из разных альбомов

Все фотографии
  Опубликованные фото с фильтрами по тегам
```

Главная сущность - альбом. Альбом работает как папка или пленка: например, `Film 060` с 36 фотографиями или цифровая серия с 30 фотографиями.

Один альбом может относиться к нескольким сетам. Сет - это большая публичная/редакторская секция из альбомов: например, `Bangkok`, `Mountains`, `Milky Way Timelapse`.

Теги должны быть из единого справочника, чтобы не плодить варианты вроде `Bangkok`, `bangkok`, `BKK`.

Отдельно будут коллекции: выбранные фотографии из разных альбомов. Например, `Selected Work`, `Portfolio 2026`, `Best of Film`. Коллекция не копирует файлы, а ссылается на уже загруженные фото.

У альбомов и фотографий будут теги. Фотография автоматически наследует теги своего альбома, а дополнительно ей можно назначить собственные теги. У сетов тегов пока не будет: сет сам является большим объединением альбомов.

Практическое правило:

- сет содержит альбомы;
- альбом содержит фотографии;
- коллекция содержит отдельные фотографии;
- фото-тег описывает конкретный кадр;
- режим "все фотографии" позволяет фильтровать кадры по тегам и унаследованным тегам альбома.

## Файлы фотографий

Для каждой фотографии планируются такие уровни:

```txt
thumb       - маленькое превью, примерно до 100 KB
display     - версия для просмотра на сайте, примерно до 1 MB
expanded    - увеличенная web-версия, примерно 2-3 MB
downloadJpeg - публичный файл для скачивания/открытия, примерно 3-4 MB
sourceJpeg  - загруженный большой JPEG, обычно 5-30 MB
master      - RAW/RAF/TIFF/панорама, только для избранных
```

Большой загруженный JPEG по умолчанию не должен просто так отдаваться посетителю. Он хранится для админки и архива. Публичный сайт в большинстве случаев работает с оптимизированными web-версиями.

Возможность открыть или скачать более крупный файл можно включать отдельно на уровне альбома или отдельной фотографии.

По умолчанию публичная кнопка download должна скачивать оптимизированный `downloadJpeg` примерно 3-4 MB, а не приватный `sourceJpeg`. Это решение должно быть настройкой в админке: можно будет переключить на `expanded` или выключить.

RAW/RAF/TIFF в текущей версии остаются private-only и доступны только владельцу через админку. Позже можно добавить скачивание исходников для выбранных пользователей.

## Публикация

Желаемый flow:

```txt
загрузил фото
  -> система создала web-версии
  -> фото попали в draft/review
  -> владелец проверил порядок, теги, обложки, видимость
  -> владелец нажал publish
```

Публикация должна быть осознанным действием. Не надо автоматически выкладывать все сразу после upload.

На первом этапе upload делаем только для JPEG.

## Hidden, Trash, Delete

`hidden`:

- фото пропадает с публичного сайта;
- фото пропадает из публичных коллекций и режима "все фотографии";
- фото остается в админке;
- файлы остаются в R2;
- фото можно вернуть обратно.

`trash`:

- фото временно в корзине;
- можно восстановить;
- можно настроить автоудаление, например через 7 дней.
- если альбом уходит в корзину, его фотографии тоже уходят в корзину и исчезают отовсюду публично.

`delete`:

- фото и связанные файлы полностью удаляются из Cloudflare R2;
- действие необратимо после окончательного удаления.

## Обложки

По умолчанию обложкой альбома становится первая фотография.

Нужно поддержать:

- горизонтальную обложку как основную;
- квадратную обложку, полученную из горизонтальной;
- вертикальную обложку как дополнительную;
- выбор обложки из фото альбома;
- возможность загрузить отдельную обложку;
- crop для обложек.

## Цвет

Для первых web-версий лучше использовать предсказуемый web-pipeline:

- `thumb` и `display` делать в sRGB;
- `sourceJpeg` хранить как загружен, сохраняя embedded color profile, если возможно;
- GPS и чувствительный EXIF не публиковать.

Позже можно добавить расширенную настройку: сохранять P3/широкий цветовой профиль для web-версий там, где это действительно нужно.

## Cloudflare

Планируемая инфраструктура:

```txt
Cloudflare Pages     - публичный сайт
Cloudflare Access    - вход в админку для владельца
Cloudflare Workers   - API админки и загрузки
Cloudflare R2        - файлы фотографий
Cloudflare D1        - сеты, альбомы, фото, статусы, теги
Cloudflare Queues    - обработка изображений
```

Первичный доступ в админку можно закрыть через Cloudflare Access для:

```txt
Jacobjshmol@gmail.com
```

## Где жить админке

На первом этапе лучше держать админку в этом же репозитории:

```txt
/admin
  статический frontend админки
  защита Cloudflare Access
  API через Workers / Pages Functions
```

Так проще быстрее собрать рабочую систему. Если админка вырастет в отдельный большой продукт, ее можно будет вынести в отдельное приложение позже.

## Публичные данные и API

Публичный сайт должен перейти к live API на Cloudflare:

```txt
страницы и интерфейс
  -> запрашивают опубликованные сеты, альбомы, коллекции и фото
  -> API читает D1
  -> API возвращает только published/not hidden
  -> изображения грузятся из R2 public assets
```

Это позволит фильтровать фото по нескольким тегам без пересборки сайта после каждого изменения.

При этом главные страницы можно оставить очень быстрыми и легкими: статический UI + живые запросы к API.

## Seed Content

Фейковый контент лучше не смешивать с настоящим архивом.

Для разработки можно сделать отдельные demo/fixture записи, которые легко удалить. На production лучше публиковать только реальные альбомы, даже если их сначала мало.

## Админка как следующий фокус

Следующая большая работа - админка.

Первый рабочий MVP должен позволять:

- создавать сеты;
- создавать альбомы;
- загружать JPEG-фотографии в альбом;
- сохранять порядок загрузки;
- назначать теги альбомам и фотографиям;
- скрывать и публиковать фото;
- отправлять альбомы и фото в корзину;
- восстанавливать из корзины;
- выбирать обложки;
- менять глобальные настройки обработки изображений.

Админку начинаем в этом же репозитории под `/admin`, закрываем Cloudflare Access.
